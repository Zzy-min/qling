# `qling` TUI 排队输入反馈规格（2026-06-01）

## 背景

TUI 已通过本地内存串行队列避免多个输入并发进入同一个 `AgentLoop`。但当用户在 Agent 正在执行时继续提交输入，目前没有即时反馈；用户可能误以为输入丢失、卡住，或重复提交同一内容。

## 目标

- 当输入因当前任务正在处理而进入队列时，立即输出一条轻量提示。
- 提示只展示等待数量，不展示用户输入正文，避免泄露 prompt 或 secret。
- 第一条立即处理的输入不提示“已排队”。
- 保持队列只存在于本进程内存，不写磁盘、不联网、不调用模型。
- 不改变串行执行顺序、不改变 `/goal`、`/loop`、`/statusline` 和本地历史行为。

## 非目标

- 不新增队列管理命令。
- 不实现队列取消、编辑、重排。
- 不持久化队列。
- 不在提示里打印用户输入内容。

## 行为

- 若 `SerialInputQueue` 当前正在处理或已有待处理项，新提交输入触发 `onQueued({ pendingCount })`。
- `pendingCount` 表示当前等待处理的队列长度，包含刚入队的输入，不包含正在运行的输入。
- `StreamingREPL` 收到队列通知后输出 `输入已排队，等待处理: N`。
- 队列通知失败不得影响入队和处理。

## 验收

- 单测覆盖第一条输入不触发排队通知。
- 单测覆盖忙时第二/第三条输入触发通知且数量正确。
- 单测覆盖通知回调拿不到或不需要显示输入正文。
- `StreamingREPL` 的提示文案不包含用户输入正文。
- `npm run build`、目标单测和 `npm run ci:check` 通过。
